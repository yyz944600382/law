## 项目概述

Law Agent 是一个**多智能体(multi-agent)法律推理系统**,模拟法庭辩论过程。

- 用户输入一个**完整的法律场景**(案件事实描述),系统视为自洽、无歧义的完整输入。
- 通过多个 Agent 协作进行对抗式辩论,最终由「法官」给出**裁决 + 说理 + 引用法条**。
- **当前阶段不支持多轮对话**:用户无法追问、补充背景或与法官交互。用户输入即终结输入。

## 系统架构

系统由 **4 个 Agent** 组成,以法官 Agent 为中心编排:

| Agent | 角色 | 核心职责 |
|-------|------|----------|
| **法官 Agent (Judge)** | 主 Agent / Orchestrator | 系统入口。理解场景 → 拆解出正反两方立场 → 分派给两位律师 → 控制辩论节奏 → 综合双方论点产出最终裁决 |
| **律师 Agent A (Plaintiff)** | 一方代理人 | 代表其中一方(如原告)。以指定立场 prompt 调用法律条文 Agent 检索依据并发言 |
| **律师 Agent B (Defendant)** | 对立方代理人 | 代表对立一方(如被告)。立场与 A 严格相反,同样调用法律条文 Agent 检索并发言 |
| **法律条文 Agent (LawSearch)** | 知识库 / 检索服务 | 存储法律知识,按律师传入的检索意图返回**真实、可溯源**的相关法条 |

### 交互流程

```
用户输入(完整法律场景)
        │
        ▼
   法官 Agent ── 理解场景,拆解两方立场
        │
        ├──► 律师 A(立场X)──┐
        │                     ├──► 法律条文 Agent(共享检索)
        └──► 律师 B(立场Y)──┘
                 ▲   ▲
                 │   └─── 多轮对抗发言(辩论)─────┐
                 └────────────────────────────────┘
        │
        ▼
   法官 Agent ── 综合双方发言,产出裁决
        │
        ▼
   返回用户(结构化结果)
```

1. 用户输入完整法律场景 → 进入法官 Agent。
2. 法官 Agent 理解场景,拆解出两方对立立场,分别交给律师 A / B。
3. 律师 A、B 各自带**不同立场 prompt** 调用法律条文 Agent,获取适用法条。
4. 律师 A、B 进行**多轮对抗发言**(辩论)。
5. 法官 Agent 收集双方发言,综合得出裁决,返回用户。

### 架构设计约定

- **法律条文 Agent 无状态、可复用**:它本身不持有立场,仅依据律师传入的检索意图返回法条,由律师 A、B 共享调用。
- **立场严格隔离**:两位律师的 prompt 必须严格对立,禁止立场漂移。
- **辩论有界**:律师对抗必须设置**最大轮数**(建议可配置,默认如 3 轮),防止无限循环。
- **裁决结构化**:法官输出应包含 —— 裁决结论、事实认定、双方主要论点摘要、引用法条。

## 协作流程(文档驱动)

本项目以**文档驱动协作**,所有需求与经验沉淀在 `docs/` 下:

| 目录 | 用途 | 谁写 |
|------|------|------|
| `docs/specs/` | 需求规格。你写简略 spec,我据此理解需求并实现 | 你写,我可辅助细化 |
| `docs/devlog/` | 开发日志。记录每次迭代 / 对话的关键结论与决策 | 我维护(每轮迭代追加) |
| `docs/lessons/` | 经验沉淀。问题 → 根因 → 解决方案,可检索 | 解决问题后我写入 |

### 工作循环
1. 你在 `docs/specs/` 写一个简略 spec(参考 `docs/specs/README.md` 模板)。
2. 我理解 spec,确认无误后实现(走下方分支 / 测试规范)。
3. 实现完成后,我在 `docs/devlog/` 追加本次迭代记录。
4. 若解决了值得记录的问题,我在 `docs/lessons/` 沉淀一条经验。
5. **遇到任何问题,先检索 `docs/lessons/` 和已有 spec,避免重复踩坑。**

### 文档同步
- **CLAUDE.md 随对话持续更新**:对话中确定的新约定、新事实,及时写入本文件。
- 文档变更与代码变更一起,走同一套分支 / 提交 / 测试规范。

## 关键风险与注意事项

- **法条幻觉(最高优先级)**:法律条文 Agent 必须返回**真实、可溯源**的法条,标注出处(法律名称 + 条款号)与版本/生效时间;绝不允许 LLM 编造法条。律师引用时必须附带来源。
- **立场漂移**:对抗辩论中律师须守住己方立场,法官不得在分派阶段泄露倾向。
- **可观测性**:多 Agent 系统建议记录每轮发言、检索的法条、法官决策依据,便于排查与审计。
