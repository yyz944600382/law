## 项目概述

Law Agent 是一个**多智能体(multi-agent)法律推理系统**,模拟法庭辩论过程。

- 用户输入一个**完整的法律场景**(案件事实描述),系统视为自洽、无歧义的完整输入。
- 通过多个 Agent 协作进行对抗式辩论,最终由「法官」给出**裁决 + 说理 + 引用法条**。
- **当前阶段不支持多轮对话**:用户无法追问、补充背景或与法官交互。用户输入即终结输入。

## 系统架构

系统由 **4 个 Agent** 组成,以法官为中心编排:

| Agent | 角色 | 实现方式 | 核心职责 |
|-------|------|----------|----------|
| **法官 (Judge)** | 主控 / Orchestrator | **JudgeEngine(纯 TS 状态机)+ LLM 结构化输出** | 入口。理解场景 → 拆解两方立场 → 派发律师 → 控辩论节奏 → 综合裁决;JudgeEngine 校验裁决完整性 |
| **律师 A (Plaintiff)** | 一方代理人 | LLM + 检索 tool | 代表原告,检索法条并陈词 |
| **律师 B (Defendant)** | 对立方代理人 | LLM + 检索 tool | 代表被告,立场与 A 严格相反,检索法条并反驳 |
| **法律检索 (LawSearch)** | 检索服务 | 联网搜索(demo 免 key 凑合) | 按律师检索意图返回相关法条(标注临时 + 幻觉风险) |

### 庭审流程

```
用户输入(完整法律场景)
        │
   JudgeEngine(纯 TS 状态机,控流程/校验)
        │
   1. 法官 LLM 拆解 → {争议焦点, 原告立场, 被告立场}
   2. 派发两位律师(各检索法条 + 陈词)
   3. 控辩论轮次(demo 各 1 轮)
   4. 收集双方论点
   5. 法官 LLM 产 {裁决结论, 事实认定, 引用法条}
   6. JudgeEngine 校验裁决完整性(必须有引用法条、必须结构化)
        │
   结构化裁决 + 可追溯日志 → 返回用户
```

### 架构设计约定

- **法官 = 确定性 JudgeEngine + LLM**:LLM 只产结构化裁决意图,纯 TS 代码控流程 / 举证责任 / 合法性校验。这是控幻觉的底线(吸收自 clowder-ai `WerewolfEngine` 范式)。
- **法律检索无状态、可复用**:不持有立场,按律师检索意图返回,律师 A/B 共享调用。
- **立场严格隔离**:两位律师 prompt 严格对立,禁止立场漂移。
- **辩论有界**:律师对抗设置最大轮数(demo 默认各 1 轮),防无限循环。
- **裁决结构化**:法官输出 = 裁决结论 + 事实认定 + 双方论点摘要 + 引用法条(≥1 条)。

## 技术栈

- **语言**:TypeScript(**纯自写,不引 agent 框架**)
- **包管理**:npm(Node 23 自带)
- **LLM**:DeepSeek,全 agent 用 `deepseek-chat`(OpenAI 兼容端点 `https://api.deepseek.com/v1`,key 取自 `DEEPSEEK_API_KEY`)
- **运行**:`npx tsx` 直接运行(**无 Docker**)
- **法律检索(demo)**:联网搜索(免 key DuckDuckGo),**临时方案、有幻觉风险**,后续换可控法条库
- **日志**:结构化 JSONL(每轮发言 / 检索法条 / 裁决),写 `logs/`,可追溯
- **测试**:vitest,**禁止 mock**(真实 DeepSeek + 真实联网 + 端到端)
- **关键依赖**:zod(结构化校验)、tsx、vitest、原生 fetch

### 设计参考
本项目借鉴本地 `~/Agent/clowder-ai`(产品级平台,**不直接复用其代码**)的几个设计:`LlmAIProvider`(程序内直连 LLM API)、`WerewolfEngine`(LLM 出意图 + 纯代码裁决)、`a2a-mentions`(消息路由)。详见 `docs/plans/0001-ts-rewrite-plan.md`。

- **禁止在主分支 `main` 上直接开发。**

- **一个逻辑变更一个 commit,避免混合无关改动。**

## 协作流程(文档驱动)

本项目以**文档驱动协作**,所有需求与经验沉淀在 `docs/` 下:

| 目录 | 用途 | 谁写 |
|------|------|------|
| `docs/specs/` | 需求规格。你写简略 spec,我据此理解需求并实现 | 你写,我可辅助细化 |
| `docs/devlog/` | 开发日志。每次迭代 / 对话的关键结论与决策 | 我维护 |
| `docs/lessons/` | 经验沉淀。问题 → 根因 → 解决方案,可检索 | 解决问题后我写 |
| `docs/plans/` | 实现计划。每个大特性的规划文档 | 我写,你审批 |

### 工作循环
1. 你在 `docs/specs/` 写简略 spec(参考 `docs/specs/README.md` 模板)。
2. 我理解 spec → 大特性先写 `docs/plans/` 计划 → 你审批 → 实现。
3. 实现完成后,我在 `docs/devlog/` 追加迭代记录。
4. 解决的问题沉淀到 `docs/lessons/`。
5. **遇到任何问题,先检索 `docs/lessons/` 和已有 spec/plan,避免重复踩坑。**

### 文档同步(强制 · 高优先级)
- **及时更新文档是硬性要求,不得延后或遗漏**:对话中一旦确定新约定、新事实、新决策或新经验,必须**当场**写入对应文档:
  - 新约定 / 新事实 / 架构变更 → `CLAUDE.md`
  - 本次迭代的结论与决策 → `docs/devlog/`
  - 解决的问题与经验 → `docs/lessons/`
  - 需求确认 / 状态变化 → `docs/specs/`
  - 大特性规划 → `docs/plans/`
- 文档与代码一起,走同一套分支 / 提交 / 测试规范。

## 关键风险与注意事项

- **法条幻觉(最高优先级)**:法律检索必须返回**真实、可溯源**的法条,标注出处与版本;绝不允许 LLM 编造法条。律师引用须附来源。
  > ⚠️ demo 阶段法律检索用联网搜索凑合,此底线暂被放宽;正式版必须换可控法条库。JudgeEngine 会对裁决做结构化校验(必须含引用法条)兜底。
- **立场漂移**:律师须守住己方立场;法官拆解时不泄露倾向。
- **可观测性**:每轮发言、检索的法条、法官决策都记入 `logs/`,便于排查与审计。
